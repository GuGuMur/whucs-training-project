## Key Learnings

- **Project:** whucs-training-project
- **Project docs:** Requirements deliverables for this course project can be placed under `report/requirements/`; the main project theme is a large-model intelligent file management and agent collaboration platform combining file management, RAG, visual workflows, RBAC, and team collaboration.
- **Open Design config:** For this project, Open Design integration is project-level: root `DESIGN.md` supplies the active design system and `skills/` supplies committed project skills; the system `od` binary may be GNU coreutils, so verify the Open Design CLI before running `od mcp install codex`.
- **Frontend MVP pattern:** The workbench root route uses Vue 3 + Vite + Pinia + Naive UI + UnoCSS. `WorkspaceView.vue` composes feature components from `frontend/src/components/workspace/`, chooses desktop/mobile shells from `frontend/src/layouts/`, and gets typed demo/client data from `frontend/src/client/workspace.ts` through `frontend/src/stores/workspace.ts`.
- **Frontend architecture boundary:** Keep frontend top-level `src/` limited to `assets/`, `auth/`, `client/`, `components/`, `composables/`, `layouts/`, `plugins/`, `router/`, `stores/`, `views/`, `App.vue`, and `main.ts`. `client/` is the future `@hey-api/openapi-ts` generation target after backend OpenAPI work; `auth/` remains a separate permission/session module linked to `client/`.
- **OpenAPI client generation:** `frontend/openapi-ts.config.ts` reads `src/client/openapi/workspace.openapi.json` and writes generated SDK/types to `src/client/generated/`; `pnpm generate:client` runs backend export with `uv run --no-project --with fastapi --with python-multipart` before `openapi-ts`.
- **Generated client integration:** `frontend/src/client/workspace.ts` is the hand-written page adapter and demo-data owner, but live API calls should go through `@/client/generated` SDK functions with auth headers from `src/auth/`.
- **Auth boundary:** `frontend/src/auth/` owns session persistence and auth header helpers; `frontend/src/stores/auth.ts` owns generated-client login/register/session restore state; routes use auth guard logic without moving permission state into `client/` or `views/`.
- **Refresh-token boundary:** Backend auth tokens carry a token kind, and the MVP enforces refresh tokens only on `/auth/refresh` while protected resources require access tokens. Frontend refresh calls must go through the generated SDK from `src/stores/auth.ts` and persist the rotated session via `src/auth/`.
- **Profile maintenance boundary:** `PATCH /api/v1/users/me` is the FR-U04 profile contract. Backend service validates duplicate emails and records `user.update_profile`; frontend calls `updateMeApiV1UsersMePatch` only from `stores/auth.ts`, while the page is `views/ProfileView.vue`.
- **Login lockout boundary:** FR-U05 is implemented in the in-memory auth service as a 5-failure, 5-minute lockout with structured `ACCOUNT_LOCKED` details and audit events. Frontend displays this through `stores/auth.ts`; `LoginView.vue` remains a thin presenter.
- **File lifecycle boundary:** File download/delete routes are backend file-domain actions with audit events. Frontend calls generated SDK functions through `src/client/workspace.ts`, keeps token helpers in `src/auth/`, owns state in `stores/workspace.ts`, and leaves `FileWorkbench.vue` as a Naive UI presenter that emits row actions.
- **Recycle-bin boundary:** FR-F09 soft delete keeps deleted file bytes and versions in backend memory, removes the file from active list/download/share/version paths, exposes restore via generated OpenAPI client adapters, and stores frontend recycle-bin state/actions in `stores/workspace.ts`.
- **File search/upload boundary:** Existing backend list/upload contracts are surfaced through generated SDK adapters in `src/client/workspace.ts`; filters and upload state live in `stores/workspace.ts`, `FileWorkbench.vue` remains a presenter, and upload controls are isolated in `FileUploadPanel.vue`.
- **File reparse boundary:** File reparse uses v2 generated SDK adapter `reparseWorkspaceFile()` -> `stores/workspace.ts` `reparseFile()` -> snapshot row replacement. Presenters emit `reparse-file` only; do not call `fetch()` or `location.reload()` from file components. The v2 DB service stores uploaded bytes in SHA-256-keyed local object files under `backend/.data/file-contents` and uses `_FILE_CONTENTS` only as an in-process cache, so reparse/download survive request boundaries and backend restarts after new uploads.
- **Legacy reparse data gap:** Files uploaded before SHA-256 durable content storage may have DB metadata but no `backend/.data/file-contents/<sha>` object and no `file_versions` recovery row. Reparse cannot succeed for those legacy rows; surface this as a typed missing-content error or parse error in the API/UI, and require reupload/edit/migration to restore bytes.
- **DB-backed v2 file manager boundary:** `frontend/src/components/files/*` are presenters that emit preview/edit/version/permission/reparse events; `FileManagerView.vue` wires those to `stores/workspace.ts`; `frontend/src/client/workspace.ts` maps generated SDK calls. V2 file content preview/edit uses `GET/PATCH /api/v2/files/{file_id}/content`; text edits create new `FileVersion` rows, update durable SHA content, set `parse_status` back to `queued`, and require explicit reparse to re-index.
- **Generated SDK runtime import gotcha:** If Vite dev reports that `src/client/generated/index.ts` does not provide a newly generated SDK export even after regeneration, import that SDK function directly from `@/client/generated/sdk.gen` in the hand-written adapter to bypass the barrel re-export at runtime.
- **File search time-range boundary:** FR-F07 updated-time filtering uses backend `updated_from`/`updated_to` query parameters and frontend `updatedFrom`/`updatedTo` filter fields. Keep the mapping in `src/client/workspace.ts`, filter state normalization in `stores/workspace.ts`, and the visible toolbar controls in `FileWorkbench.vue`.
- **File annotation boundary:** US-T03/FR-C05 file annotations use backend `/files/{file_id}/annotations`, `/annotations/{annotation_id}/replies`, and delete routes. Frontend maps generated snake_case contracts in `src/client/workspace.ts`, stores annotation lists under `fileAnnotationsById` in `stores/workspace.ts`, and keeps discussion UI isolated in `FileAnnotationPanel.vue` composed by `FileWorkbench.vue`.
- **Notification inbox boundary:** FR-C07 notifications use backend `/api/v1/notifications` and `/api/v1/notifications/{notification_id}/read`. Backend unread counts derive from per-user notification records; frontend maps generated contracts in `src/client/workspace.ts`, stores inbox/read state in `stores/workspace.ts`, and keeps inbox UI isolated in `NotificationInboxPanel.vue` composed by `TeamAuditPanel.vue`.
- **RAG conversation boundary:** V2 RAG Q&A persists multi-turn state in `knowledge_conversations`, `knowledge_messages`, and `knowledge_citation_snapshots`; `POST /api/v2/qa/query` returns the persisted assistant `message_id`, `GET /api/v2/knowledge-bases/{kb_id}/conversations` lists summaries, and `GET /api/v2/conversations/{conversation_id}` returns messages with citation snapshots that survive later KB file changes.
- **RAG conversation deletion boundary:** `DELETE /api/v2/conversations/{conversation_id}` deletes citation snapshots, messages, and the conversation after KB access validation. Frontend routes it through `deleteKnowledgeConversation()` in `src/client/workspace.ts` and `stores/knowledge.ts`; UI deletion must clear `activeConversationId` when the deleted conversation was active.
- **RAG file-scope boundary:** Team knowledge bases may only add files whose folder is in the same team scope; personal files must be rejected with `KB_FILE_SCOPE_MISMATCH`. Personal knowledge bases should not add team files. Frontend `stores/knowledge.ts` filters `indexedFiles` by active KB scope so the picker does not offer invalid files.
- **RAG pipeline boundary:** Retrieval and grounded answer composition live in `backend/app/services/rag_pipeline.py`; `WorkspaceServiceDB` keeps public methods, access checks, conversation persistence, and FAISS cache invalidation. `QAResponse.error_code` can be `KB_EMPTY`, `KB_FILE_NOT_INDEXED`, `KB_NO_MATCH`, or `None`, and those error cases still persist the user/assistant messages.
- **Agent tool-flow boundary:** V2 tool flow uses `ToolRegistry` and `AgentExecutor`; `/api/v2/tools` exposes `calculator`, `course_lookup`, `file_content_search`, and `python_data`, while `/api/v2/agents/tasks` returns transparent `understand/plan/call/observe/answer` phases with `input_json`, `output_json`, `result_view`, and `needs_clarification` plus `/agents/tasks/{task_id}` detail and `/continue` resume routes.
- **Frontend RAG/agent adapter boundary:** `frontend/src/client/workspace.ts` exposes camelCase typed adapters for KB delete/batch add/batch remove/reindex/conversations, tool catalog, and agent detail/continue, importing newer generated SDK functions directly from `@/client/generated/sdk.gen` when needed. `frontend/src/stores/knowledge.ts` owns scoped KB metadata, batch membership, conversations/messages, active conversation id, and reindex loading state.
- **Frontend agent state boundary:** `frontend/src/stores/agent.ts` owns tool catalog, active task, task history, execution steps, clarification question, result view, loading, and error state. Workflow pages should consume this store and `components/workflow/AgentTaskComposer.vue`, `ToolCatalogPanel.vue`, `AgentExecutionTimeline.vue`, and `ToolResultViewer.vue` instead of direct `fetch()` calls for agent/tool execution.
- **File move/copy/version boundary:** FR-F01/FR-F05 lifecycle operations now use backend `PATCH /files/{file_id}`, `POST /files/{file_id}/copy`, `GET /files/{file_id}/versions`, and restore routes. Frontend maps generated snake_case contracts in `src/client/workspace.ts`, owns loading state in `stores/workspace.ts`, and keeps the lifecycle UI isolated in `FileLifecyclePanel.vue`.
- **Folder CRUD boundary:** FR-F02 folder create/tree/move/delete is now a backend/OpenAPI/generated-client/store/UI slice. Components emit camelCase payloads (`parentId`), `src/client/workspace.ts` maps them to OpenAPI `parent_id`, and `stores/workspace.ts` owns the tree, active folder, folder options, and loading state.
- **Folder move rule:** Backend folder moves cannot cross personal/team scope and cannot target self/descendants. Frontend move options should filter by same scope and exclude the active folder subtree before emitting updates.
- **Permission rule boundary:** Resource ACL rules use backend `/api/v1/permissions/rules` and generated SDK calls through `frontend/src/client/workspace.ts`; Pinia owns `permissionRules` state/actions, `PermissionRulesPanel.vue` is the presenter, and `WorkspaceView.vue` only wires handlers.
- **ACL evaluation rule:** Phase 20 keeps ACL rules in memory but enforces deny precedence, inherited folder rules, and file-level overrides through the shared file/folder read-write helpers established in Phase 19.
- **Local API proxy boundary:** Vite development should proxy same-origin `/api` calls to `http://127.0.0.1:8000`; this keeps the generated client on `baseUrl: '/'` while connecting to the FastAPI backend during local development.
- **Testing pattern:** When mounting Naive UI workspace views in Vitest, install the Naive UI plugin and wrap the tested component in `NConfigProvider`; `@pinia/testing` needs `createSpy: vi.fn` in this setup.
- **UnoCSS gotcha:** Use arbitrary unitless line-height syntax (`leading-[1.65]`) for design line-height values. Classes like `leading-1.65` can generate tiny rem line heights and visually overlap text.
- **Responsive table gotcha:** Naive UI `NDataTable` can expand the mobile page if its columns require a wider table. Use `scroll-x` and constrain card/grid wrappers with `min-w-0 overflow-hidden`.
- **Mobile design gotcha:** For DESIGN.md phone behavior, do not stop at horizontal scrolling for operational tables. Keep `NDataTable` on wider screens, but render grouped mobile rows for files and other daily-use entities.

- [2026-07-10] Do not implement file-manager reparse as direct component `fetch()` plus `location.reload()`; route it through the generated client adapter and Pinia store so the returned `FileItem` replaces the current snapshot row.
- [2026-07-10] When adding a brand-new generated SDK function to `frontend/src/client/workspace.ts`, verify the Vite runtime path. The generated barrel may type-check but still fail in dev HMR; direct `sdk.gen` imports are acceptable for affected functions.
- [2026-07-10] Do not treat process memory as the source of truth for v2 uploaded file content. Reparse, download, and knowledge indexing must read through durable content storage keyed by file SHA-256, with memory only as a cache.
- [2026-07-10] Do not derive DB file primary keys only from content SHA without collision handling; repeated uploads of identical bytes must generate a unique fallback id after checking the repository.
- [2026-07-10] Do not return a silent 200 from v2 reparse when original file bytes are missing for a legacy DB row; expose a typed missing-content failure so the frontend can tell the user to reupload or edit the file.
- [2026-07-11] Do not use the v1 `auth_headers()` helper for v2 API tests; v2 endpoints validate DB-backed users, so v2 tests must register through `/api/v2/auth/register` via `auth_headers_v2()`.
- [2026-07-11] Do not rely on prompt history alone to prove multi-turn RAG coherence; if the user asks about prior turns, use persisted conversation messages for deterministic context handling because a live LLM may ignore or down-rank appended history.
